## 배경 및 목적

현재 Flowrit에 알림 시스템이 없다. 고객이 수정 요청을 제출해도, 신규 의뢰가 접수되어도, 담당 프로젝트의 마감이 임박해도 사업자는 대시보드를 직접 열어 확인해야 한다. Resend가 이미 세팅되어 있으나 팀 초대 이메일 하나에만 사용된다.

알림 시스템의 핵심 목적은 "Flowrit를 열지 않아도 중요한 일이 생기면 알 수 있다"는 것이다. 이 스펙은 이메일 알림을 기반으로 하며, 인앱 알림(알림 벨)은 선택적 후속 기능으로 별도 정의한다.

---

## 사용자 스토리

- **US-401**: 사업자로서, 고객이 수정 요청을 제출하면 이메일로 즉시 알림을 받고 싶다.
- **US-402**: 사업자로서, 고객이 신규 의뢰 접수 폼을 제출하면 이메일로 알림을 받고 싶다.
- **US-403**: 담당 프로젝트의 단계가 팀원에 의해 변경되면 이메일로 알림을 받고 싶다.
- **US-404**: 사업자로서, 마감일 1일 전에 해당 프로젝트를 이메일로 리마인드 받고 싶다.
- **US-405**: 사업자로서, 특정 이벤트의 이메일 알림을 설정 페이지에서 켜거나 끌 수 있기를 원한다 (001-settings의 FR-209와 연동).
- **US-406**: 사업자로서, 앱 내 알림 아이콘을 통해 읽지 않은 중요 이벤트를 빠르게 확인하고 싶다.

## 기능 요구사항

### 이메일 알림

- **FR-401**: 고객이 `/intake/[workspaceSlug]`에서 신규 의뢰를 제출하면, 해당 워크스페이스의 Owner와 `notify_new_inquiry` 설정이 활성화된 멤버에게 이메일을 발송한다.
  - 이메일 내용: 제출자 이름, 연락처, 의뢰 내용 요약(100자), 대시보드 바로 가기 링크
- **FR-402**: 고객이 `/p/[token]/revision`에서 수정 요청을 제출하면, 해당 프로젝트의 담당자(`assigneeId`)와 `notify_revision_submitted` 설정이 활성화된 Owner에게 이메일을 발송한다. 담당자가 없으면 Owner에게만 발송한다.
  - 이메일 내용: 프로젝트명, 수정 요청 내용, 첨부 파일 수, 프로젝트 바로 가기 링크
- **FR-403**: 프로젝트 단계가 변경될 때, 해당 프로젝트의 담당자가 변경을 수행한 사용자 본인이 아닌 경우 담당자에게 이메일을 발송한다. `notify_stage_changed` 설정이 비활성화된 경우 발송하지 않는다.
  - 이메일 내용: 프로젝트명, 이전 단계 → 새 단계, 변경한 팀원 이름, 프로젝트 바로 가기 링크
- **FR-404**: 마감일(`dueDate`)이 설정된 프로젝트 중 마감 24시간 전(±1시간 오차 허용)에 해당하는 진행 중인 프로젝트의 담당자(없으면 Owner)에게 리마인더 이메일을 발송한다. `notify_deadline_soon` 설정이 비활성화된 경우 발송하지 않는다.
  - 이메일 내용: 프로젝트명, 고객명, 마감일, 미완료 수정 요청 건수, 프로젝트 바로 가기 링크

### 마감일 리마인더 스케줄링

- **FR-405**: 마감일 리마인더(FR-404)는 Vercel Cron Job을 통해 매 정시에 실행한다. 실행 간격은 1시간이며, 해당 시간 기준 23~25시간 후 마감인 프로젝트를 조회한다.
- **FR-406**: 동일 프로젝트·동일 마감일에 대해 리마인더가 중복 발송되지 않도록 `TimelineEvent`에 `DEADLINE_NOTIFIED` 이벤트 타입을 기록하고 이미 기록된 경우 발송을 건너뛴다.

### 인앱 알림 센터

- **FR-407**: 사이드바 또는 헤더에 알림 벨(🔔) 아이콘을 추가한다. 읽지 않은 알림이 있으면 빨간 뱃지(최대 99+)를 표시한다.
- **FR-408**: 알림 벨 클릭 시 드롭다운 패널이 열리며, 최근 알림 20건을 최신순으로 표시한다. 각 알림은 이벤트 유형, 요약, 관련 프로젝트/의뢰 링크, 시간을 포함한다.
- **FR-409**: 알림 패널을 열면 표시된 알림이 자동으로 "읽음" 처리된다.
- **FR-410**: 알림은 `Notification` 모델로 저장된다. 필드: `id`, `userId`, `workspaceId`, `type`, `title`, `body`, `href`, `isRead`, `createdAt`.
- **FR-411**: 알림을 생성하는 이벤트는 FR-401~FR-404와 동일한 이벤트이며, 이메일 발송과 동시에 알림 레코드도 생성된다. 알림 대상자는 이메일 수신자와 동일하다.

## 비기능 요구사항

- **NFR-401**: 이메일 발송 실패는 Server Action 또는 Cron Job 실행 흐름에서 `try/catch`로 처리하며, 발송 실패가 핵심 기능(수정 요청 저장, 단계 변경 등)을 실패시키지 않는다.
- **NFR-402**: 마감일 리마인더 Cron Job의 실행 횟수는 무료 Vercel Hobby plan 기준 분당 1회 이하이므로 1시간 단위로 설정한다. 실행 시간은 UTC 기준이며 한국 시간 사용자를 위한 보정은 이 스펙의 범위 외다.
- **NFR-403**: `Notification` 레코드는 90일 후 자동 삭제되지 않는다. 삭제 정책은 향후 플랜 기능에서 별도 정의한다.
- **NFR-404**: 이메일 HTML 템플릿은 인라인 스타일로 구성하여 주요 이메일 클라이언트(Gmail, Outlook)에서 정상 렌더링되도록 한다.

## 수용 기준

- **SC-401** (`FR-401`): 테스트 환경에서 인테이크 폼을 제출하면 워크스페이스 Owner 이메일로 신규 의뢰 알림이 수신된다.
- **SC-402** (`FR-402`): 고객 포털에서 수정 요청을 제출하면 해당 프로젝트 담당자에게 이메일이 발송된다. 담당자 미지정 시 Owner에게 발송된다.
- **SC-403** (`FR-403`): 팀원 A가 프로젝트 단계를 변경하면 담당자 B(≠A)에게 이메일이 발송된다. A가 담당자이면 발송되지 않는다.
- **SC-404** (`FR-404`, `FR-405`, `FR-406`): 마감일이 약 24시간 후인 프로젝트가 존재할 때 Cron Job이 실행되면 담당자에게 리마인더 이메일이 발송되고, 동일 프로젝트에 대해 두 번째 Cron 실행에서는 발송되지 않는다.
- **SC-405** (`FR-209` 연동, `FR-401`): 설정 페이지에서 `notify_new_inquiry`를 끈 사용자는 신규 의뢰가 접수되어도 이메일을 받지 않는다.
- **SC-406** (`FR-407`, `FR-408`, `FR-409`): 알림 벨 아이콘에 읽지 않은 알림 수가 표시된다. 패널을 열면 알림 목록이 보이고, 닫은 후 다시 열면 뱃지가 사라진다.

## 범위 외

- 카카오톡 알림톡·SMS 발송
- 웹 푸시 알림(브라우저 Push API)
- 알림 구독 세분화(예: 특정 고객의 이벤트만 받기)
- 이메일 보낸 사람 도메인 커스터마이징(DKIM/SPF 설정)
- 알림 내역 페이지(별도 전용 페이지 — 드롭다운 패널로 갈음)
- Slack·Teams 웹훅 연동
