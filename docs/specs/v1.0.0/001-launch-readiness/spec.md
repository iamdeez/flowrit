# Spec: 1.0.0 런칭 준비 및 제품 완성도 개선

> Branch: 001-launch-readiness | Date: 2026-06-23 | Version: v1.0.0

## 배경 및 목적

Flowrit은 고객 의뢰 접수, 주문서, 프로젝트 진행, 수정 요청, 납품, 팀 협업, 결제 기능을 갖춘 상태로 배포되었다. 1.0.0 전에는 기능 추가보다 실제 서비스처럼 보이고 운영할 수 있는 완성도가 필요하다.

pre-launch 단계에서 이미 완료된 항목:
- 비밀번호 초기화(`/forgot-password`, `/reset-password/[token]`)
- 개인정보처리방침(`/privacy`) 및 이용약관(`/terms`) 페이지
- DB 인덱스 추가 및 마이그레이션 자동화(`package.json` build script)
- API 입력값 길이 제한 및 honeypot 필드
- Sentry 에러 트래킹 설치 및 설정 파일 구성(`sentry.*.config.ts`)
- Upstash Redis 기반 rate limiting(`lib/ratelimit.ts`) — intake/webhook 엔드포인트 적용 완료
- CI 워크플로우(`.github/workflows/ci.yml`) — lint/typecheck/test 포함

본 스펙은 이 기반 위에서 UI/UX 완성도, 운영 모니터링, E2E 검증을 달성한다. 단, PRO 구독 결제 흐름(`v0.7.0/001-nicepayments`)은 별도 스펙으로 진행 중이며 본 스펙 구현 전 완료되어야 한다.

이 스펙은 다음 목표를 달성한다.

- 신규 사용자가 Flowrit의 가치를 빠르게 이해하고 첫 업무 흐름을 완료한다.
- 주요 업무 페이지의 UI/UX를 일관된 제품 경험으로 정리한다.
- 오류, 빈 상태, 로딩 상태, 권한 상태에서 사용자가 다음 행동을 이해한다.
- 운영 중 문제를 조기에 감지하고 Discord로 알림을 받을 수 있다.
- 1.0.0 배포 전 검증 기준과 E2E 시나리오를 갖춘다.

## 사용자 스토리 (User Stories)

- **US-001**: 신규 사용자로서, 가입 직후 무엇부터 설정해야 하는지 안내받고 첫 주문서·프로젝트 흐름을 빠르게 경험하고 싶다.
- **US-002**: 작업자로서, 대시보드에서 오늘 처리해야 할 주문서, 수정 요청, 마감 임박 프로젝트를 우선순위대로 보고 싶다.
- **US-003**: 고객으로서, 주문서와 고객 포털 화면에서 신뢰할 수 있는 서비스라는 인상을 받고 제출 후 다음 단계를 이해하고 싶다.
- **US-004**: 워크스페이스 OWNER로서, 결제·설정·웹훅·팀 관리 화면에서 현재 상태와 필요한 액션을 명확히 알고 싶다.
- **US-005**: 운영자로서, 배포 후 에러, Cron 실패, 결제 실패, 이메일 실패 등 중요한 문제를 Discord로 즉시 받고 싶다.
- **US-006**: 개발자로서, 1.0.0 배포 전 핵심 플로우가 자동 테스트와 체크리스트로 검증되기를 원한다.

## 기능 요구사항 (Functional Requirements)

- **FR-001**: 랜딩, 인증, 온보딩 화면은 Flowrit의 핵심 가치를 일관된 문구와 시각 체계로 전달해야 한다.
- **FR-002**: 가입 직후 사용자는 워크스페이스 기본 설정, 주문서 링크 확인, 샘플 프로젝트 또는 첫 프로젝트 생성으로 이어지는 온보딩 흐름을 제공받아야 한다.
- **FR-003**: 주요 빈 상태 화면은 단순한 "데이터 없음" 대신 다음 행동 버튼과 상황 설명을 제공해야 한다.
- **FR-004**: 대시보드는 오늘의 우선순위, 업무 파이프라인, 최근 활동, 주요 지표를 한 화면에서 읽기 쉽게 보여야 한다.
- **FR-005**: 프로젝트 목록과 프로젝트 상세는 상태, 고객, 마감일, 수정 요청, 납품물, 고객 공유 링크의 관계를 명확히 보여야 한다.
- **FR-006**: 주문서 관리 화면은 접수된 주문을 검토하고 프로젝트로 전환하는 흐름을 짧고 명확하게 제공해야 한다.
- **FR-007**: 고객 공개 화면(`/order/[workspaceSlug]`, `/intake/[workspaceSlug]`, `/p/[token]`)은 고객이 제출 또는 확인 후 다음 단계를 이해하도록 안내해야 한다.
- **FR-008**: 설정 화면은 프로필, 워크스페이스, 주문서, 웹훅, 알림, 결제, 위험 작업 영역을 명확한 정보 구조로 제공해야 한다.
- **FR-009**: 결제/구독 UI는 FREE/PRO 상태, 다음 결제일, 결제 실패, 취소 예약 상태를 사용자가 오해하지 않도록 표시해야 한다.
- **FR-010**: 전역 로딩, 에러, 404, 권한 없음, 폼 제출 중 상태는 Flowrit 브랜드와 일관된 메시지와 복구 액션을 제공해야 한다.
- **FR-011**: 운영 알림 시스템은 production 환경의 중요 이벤트를 Discord Webhook으로 전송해야 한다.
- **FR-012**: 운영 알림은 민감 정보를 포함하지 않고, 같은 원인의 반복 알림을 과도하게 보내지 않아야 한다.
- **FR-013**: Health Check API는 애플리케이션, 데이터베이스, 필수 환경 설정 상태를 확인할 수 있어야 한다.
- **FR-014**: 배포 전 체크리스트는 환경변수, DB 마이그레이션, Cron, 이메일, 파일 업로드, 결제, 웹훅, 도메인 상태를 검증해야 한다.
- **FR-015**: Playwright E2E는 가입/온보딩, 주문서 접수, 프로젝트 전환, 고객 포털 수정 요청, 결제 모달 진입, 모바일 공개 폼을 검증해야 한다.
- **FR-016**: 1.0.0 작업 완료 후 context.md와 infra.md는 실제 구현된 UI/운영 변경 사항을 반영해야 한다.
- **FR-017**: 로그인 흐름은 동일 IP의 반복 실패 시도를 제한하여 brute-force 공격을 방어해야 한다.
- **FR-018**: 나이스페이먼츠 전환 후 사용하지 않는 기존 토스페이먼츠 SDK 의존성은 제거되어야 한다.

## 비기능 요구사항 (Non-Functional Requirements)

- **NFR-001**: 모든 UI 개선은 기존 워크스페이스 데이터 격리와 RBAC 경계를 유지해야 한다.
- **NFR-002**: 주요 페이지는 모바일 390px, 태블릿, 데스크톱 1440px에서 텍스트 겹침 없이 사용 가능해야 한다.
- **NFR-003**: 대시보드와 목록 페이지는 불필요한 순차 DB 요청을 피하고 병렬 조회 또는 캐시 정책을 검토해야 한다.
- **NFR-004**: 운영 알림은 production에서만 Discord로 전송해야 하며, 개발 환경에서는 로그 또는 no-op으로 처리해야 한다.
- **NFR-005**: Discord 알림에는 토큰, 세션, 이메일 본문 전문, 결제 키, 빌링키, 원본 파일 URL 등 민감 정보를 포함하지 않는다.
- **NFR-006**: Health Check API는 공개적으로 민감한 내부 상태를 노출하지 않아야 한다.
- **NFR-007**: E2E 테스트는 외부 결제 실청구 없이 테스트 가능한 범위만 자동화한다.

## 수용 기준 (Acceptance Criteria)

- **SC-001** (`FR-001`, `FR-002` 관련): 신규 가입 사용자가 온보딩에서 워크스페이스 설정과 첫 액션을 완료할 수 있다.
- **SC-002** (`FR-003` 관련): 주요 빈 상태 화면에 다음 행동 CTA가 표시된다.
- **SC-003** (`FR-004` 관련): 대시보드에서 미확인 주문, 열린 수정 요청, 마감 임박 프로젝트, 최근 활동을 한 화면에서 확인할 수 있다.
- **SC-004** (`FR-005` 관련): 프로젝트 상세에서 고객 공유 링크, 수정 요청, 납품물, 상태 변경이 시각적으로 분리되어 보인다.
- **SC-005** (`FR-006` 관련): 주문서 접수 건을 프로젝트로 전환하는 CTA가 목록에서 명확히 제공된다.
- **SC-006** (`FR-007` 관련): 공개 주문서 제출 후 성공 상태와 다음 안내가 표시된다.
- **SC-007** (`FR-008` 관련): 설정 화면의 각 섹션은 사용 목적과 현재 상태가 명확히 구분된다.
- **SC-008** (`FR-009` 관련): 결제 탭에서 FREE/PRO/결제 실패/취소 예약 상태가 각각 다른 메시지와 액션으로 표시된다.
- **SC-009** (`FR-010` 관련): 전역 error/not-found/loading 화면이 Flowrit 브랜드 톤과 복구 액션을 포함한다.
- **SC-010** (`FR-011`, `FR-012` 관련): production 중요 이벤트 발생 시 Discord로 요약 알림이 전송되고 민감 정보가 포함되지 않는다.
- **SC-011** (`FR-013` 관련): Health Check API가 정상 상태와 실패 상태를 구분 가능한 응답으로 반환한다.
- **SC-012** (`FR-014` 관련): 1.0.0 배포 전 체크리스트 문서가 생성되고 필수 운영 항목이 포함된다.
- **SC-013** (`FR-015` 관련): Playwright E2E 핵심 시나리오가 CI 또는 로컬 명령으로 실행 가능하다.
- **SC-014** (`FR-016` 관련): 구현 완료 후 context.md와 infra.md가 변경된 현재 상태와 일치한다.
- **SC-015** (`FR-017` 관련): 동일 IP에서 단시간 반복 로그인 실패가 발생하면 인증이 거부되고 사용자에게 과도한 시도 안내가 표시된다.
- **SC-016** (`FR-018` 관련): `package.json`과 lockfile에서 `@tosspayments/payment-sdk` 의존성이 제거되고 빌드가 성공한다.

## 범위 외 (Out of Scope)

- 새로운 과금 모델, 쿠폰, 환불 자동화, 세금계산서 발행
- 완전한 CRM 재설계 또는 데이터 모델 전면 개편
- 네이티브 모바일 앱
- 실시간 협업 편집 기능
- 별도 관리자 백오피스
- 고객별 커스텀 브랜딩 유료 옵션
- 외부 모니터링 SaaS의 유료 플랜 구매 결정

## 미결 사항 (Open Questions)

없음. Discord Webhook, Health Check, UI/UX 개선, E2E 검증은 1.0.0 전처리 범위에 포함한다.
