## 배경 및 목적

워크스페이스에 작업자 20명, 하루 수십~수백 건의 프로젝트가 운영되는 규모에서
현재 프로젝트 목록 페이지는 전체 데이터를 한 번에 조회하므로 성능 저하가 발생한다.
또한 일반 멤버(담당자)가 자신과 무관한 프로젝트까지 모두 볼 수 있어
집중도 저하 및 보안 우려가 있다.

## 사용자 스토리 (User Stories)

- **US-001**: 담당자(MEMBER 역할)로서, 내가 담당하는 프로젝트만 보고 싶다.
- **US-002**: 오너/어드민(OWNER·ADMIN 역할)으로서, 전체 프로젝트를 담당자 기준으로 필터링하고 싶다.
- **US-003**: 어떤 역할이든, 수백 건의 프로젝트가 있어도 페이지가 빠르게 로드되길 원한다.
- **US-004**: 프로젝트 카드에서 누가 담당하는지 한눈에 확인하고 싶다.
- **US-005**: 수정 요청 목록에서도 내 담당 프로젝트의 요청만 보고 싶다 (MEMBER).

## 기능 요구사항 (Functional Requirements)

- **FR-001**: MEMBER 역할 사용자는 `Project.assigneeId = session.user.id` 조건에 해당하는 프로젝트만 조회된다.
- **FR-002**: OWNER·ADMIN 역할 사용자는 워크스페이스 전체 프로젝트를 조회하며, 담당자 기준 필터 드롭다운을 사용할 수 있다.
- **FR-003**: 프로젝트 목록은 20건 단위로 페이지네이션한다.
- **FR-004**: 프로젝트 카드에 담당자 이름(없으면 "담당자 없음")을 표시한다.
- **FR-005**: MEMBER 역할 사용자의 수정 요청 목록(`/revisions`)은 본인 담당 프로젝트의 요청만 표시한다.

## 비기능 요구사항 (Non-Functional Requirements)

- **NFR-001**: `getProjects()`는 데이터베이스에서 최대 20건만 조회한다 (전체 스캔 금지).
- **NFR-002**: MEMBER 역할 필터는 Server Action 내부에서 적용되어야 하며, 클라이언트가 필터 조건을 우회할 수 없다.
- **NFR-003**: 역할 조회는 `WorkspaceMember` 테이블에서 1회만 수행한다.

## 수용 기준 (Acceptance Criteria)

- **SC-001** (`FR-001` 관련): MEMBER 역할 계정으로 로그인 시 프로젝트 목록에 본인이 담당자인 프로젝트만 표시된다.
- **SC-002** (`FR-001` 관련): MEMBER는 URL searchParam을 조작해도 다른 사람의 프로젝트를 볼 수 없다.
- **SC-003** (`FR-002` 관련): OWNER·ADMIN은 모든 프로젝트를 보며, 담당자 드롭다운으로 특정 멤버의 프로젝트만 필터링할 수 있다.
- **SC-004** (`FR-003` 관련): 프로젝트가 20건 초과 시 이전/다음 페이지 네비게이션이 표시된다.
- **SC-005** (`FR-004` 관련): 프로젝트 카드에 담당자 이름이 표시된다. 담당자가 없으면 표시하지 않는다.
- **SC-006** (`FR-005` 관련): MEMBER 역할로 `/revisions` 접속 시 본인 담당 프로젝트의 수정 요청만 표시된다.

## 범위 외 (Out of Scope)

- 프로젝트 상세 페이지(`/projects/[id]`)의 접근 권한 제어 (현재 workspaceId로만 검증)
- 실시간 업데이트(WebSocket)
- 프로젝트 목록 무한 스크롤
- 수정 요청 목록의 페이지네이션 UI (30건 캡으로 대응)
