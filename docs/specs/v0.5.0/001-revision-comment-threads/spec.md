## 배경 및 목적

Flowrit의 v0.5 핵심 목표는 "완결된 클라이언트 경험"이다. 현재 수정 요청(RevisionRequest)은 상태 관리(OPEN/IN_PROGRESS/DONE)만 존재하며, 작업자와 클라이언트 사이의 소통 채널이 없다. 클라이언트는 수정 요청을 제출한 뒤 처리 상황을 알 수 없고, 작업자는 수정 의도를 확인하기 위해 Flowrit 외부 채널(이메일, 메신저 등)을 이용해야 한다. 이것이 v0.5에서 해소해야 할 소통 병목이다.

본 spec은 수정 요청 단위 댓글 스레드를 신규 개발하여, 작업자와 클라이언트가 Flowrit 내에서 직접 소통할 수 있는 채널을 제공한다.

---

## 사용자 스토리

- **US-001**: 작업자(OWNER/ADMIN/MEMBER)로서, 수정 요청의 맥락을 명확히 하기 위해 해당 수정 요청에 댓글을 달고 클라이언트의 답변을 받고 싶다.
- **US-002**: 클라이언트로서, 내가 요청한 수정 사항에 대해 작업자와 대화하기 위해 로그인 없이 댓글을 달고 싶다.
- **US-003**: 작업자로서, 댓글 대화의 흐름을 유지하기 위해 기존 댓글에 답글을 달고 싶다.
- **US-004**: 클라이언트로서, 작업자가 내 댓글에 답글을 달았을 때 알 수 있도록 이메일 알림을 받고 싶다.
- **US-005**: 작업자로서, 클라이언트가 새 댓글을 남겼을 때 신속히 확인할 수 있도록 인앱 알림을 받고 싶다.

## 기능 요구사항

### 댓글 조회

- **FR-001**: 시스템은 수정 요청 ID를 기준으로 해당 수정 요청에 달린 모든 댓글을 스레드 구조(최상위 댓글 + 답글)로 반환한다.
- **FR-002**: 작업자는 대시보드(`/revisions` 또는 프로젝트 상세 페이지)에서 수정 요청의 댓글 목록을 조회할 수 있다.
- **FR-003**: 클라이언트는 고객 포털(`/p/[token]`)에서 해당 프로젝트의 수정 요청별 댓글 목록을 조회할 수 있다.

### 댓글 작성 — 내부 작업자

- **FR-004**: 인증된 작업자(OWNER/ADMIN/MEMBER)는 수정 요청에 최상위 댓글을 작성할 수 있다.
- **FR-005**: 인증된 작업자는 기존 댓글에 답글(1단계 depth)을 작성할 수 있다.
- **FR-006**: 작업자의 댓글에는 해당 작업자의 이름이 표시된다.

### 댓글 작성 — 외부 클라이언트

- **FR-007**: 클라이언트는 고객 포털에서 이름과 이메일을 입력하여 수정 요청에 최상위 댓글을 작성할 수 있다. 로그인이 필요하지 않다.
- **FR-008**: 클라이언트는 고객 포털에서 이름과 이메일을 입력하여 기존 댓글에 답글(1단계 depth)을 작성할 수 있다.
- **FR-009**: 클라이언트의 댓글에는 클라이언트가 입력한 이름이 표시된다.

### 알림

- **FR-010**: 클라이언트가 댓글을 작성하면, 해당 수정 요청이 속한 프로젝트의 assignee(또는 워크스페이스 OWNER)에게 인앱 알림이 생성된다.
- **FR-011**: 작업자가 클라이언트의 댓글에 답글을 작성하면, 해당 댓글을 작성한 클라이언트의 이메일로 알림이 발송된다. (클라이언트가 이메일을 입력한 경우에만)

### 데이터 보존

- **FR-012**: 수정 요청이 DONE 또는 REJECTED 상태로 전환되어도 댓글은 삭제되지 않고 조회 가능하다.

## 비기능 요구사항

- **NFR-001**: 클라이언트 댓글 작성 시 이름 필드는 1자 이상 100자 이하, 이메일 필드는 RFC 5321 형식이어야 한다.
- **NFR-002**: 댓글 본문은 1자 이상 2,000자 이하여야 한다.
- **NFR-003**: 클라이언트의 댓글 작성은 고객 포털 토큰(`/p/[token]`)의 유효성이 검증된 경우에만 허용된다. 토큰이 유효하지 않으면 댓글 작성이 거부된다.
- **NFR-004**: 내부 작업자의 댓글 조회 및 작성에 사용되는 모든 DB 쿼리는 `workspaceId` 범위로 scope 된다 (P-001 준수).
- **NFR-005**: 댓글 스레드 depth는 최대 1단계(최상위 댓글 + 답글)로 제한한다. 답글에 답글을 달 수 없다.
- **NFR-006**: 각 수정 요청 댓글 목록 조회 응답은 최신 작성일시 기준 오름차순으로 정렬된다.

## 수용 기준

- **SC-001** (`FR-001`, `FR-002` 관련): 작업자가 대시보드에서 수정 요청을 열면, 해당 수정 요청에 달린 모든 댓글이 최상위 댓글 — 답글 구조로 표시된다. [env:integration]

- **SC-002** (`FR-003` 관련): 클라이언트가 `/p/[token]` 페이지에서 수정 요청을 열면, 해당 수정 요청의 댓글 목록이 표시된다. [env:integration]

- **SC-003** (`FR-004`, `FR-006` 관련): 인증된 작업자가 댓글 본문을 작성하고 제출하면, 댓글이 저장되고 목록에 작업자 이름과 함께 표시된다. [env:unit]

- **SC-004** (`FR-005` 관련): 인증된 작업자가 기존 댓글에 답글을 제출하면, 해당 답글이 원댓글 하위에 표시된다. [env:unit]

- **SC-005** (`FR-007`, `FR-009` 관련): 클라이언트가 이름·이메일·본문을 입력하고 댓글을 제출하면, 댓글이 저장되고 입력한 이름이 표시된다. [env:unit]

- **SC-006** (`FR-008` 관련): 클라이언트가 기존 댓글에 답글을 제출하면, 해당 답글이 원댓글 하위에 표시된다. [env:unit]

- **SC-007** (`FR-010` 관련): 클라이언트가 댓글을 작성하면 프로젝트 assignee(또는 OWNER)에게 인앱 알림이 생성된다. [env:unit]

- **SC-008** (`FR-011` 관련): 작업자가 클라이언트 댓글에 답글을 달면, 해당 클라이언트의 이메일 주소로 알림 이메일이 발송된다. 이메일을 입력하지 않은 경우 발송되지 않는다. [env:unit]

- **SC-009** (`FR-012` 관련): 수정 요청 상태가 DONE으로 변경된 후에도 해당 수정 요청의 댓글이 조회 가능하다. [env:unit]

- **SC-010** (`NFR-001` 관련): 클라이언트가 이름을 공란으로 두거나 101자 이상 입력하면 댓글 제출이 거부된다. [env:unit]

- **SC-011** (`NFR-001` 관련): 클라이언트가 RFC 5321 형식이 아닌 이메일을 입력하면 댓글 제출이 거부된다. [env:unit]

- **SC-012** (`NFR-002` 관련): 댓글 본문이 공란이거나 2,001자 이상이면 제출이 거부된다. [env:unit]

- **SC-013** (`NFR-003` 관련): 유효하지 않은 프로젝트 토큰으로 클라이언트 댓글 작성을 요청하면 거부된다. [env:unit]

- **SC-014** (`NFR-005` 관련): 답글(depth=1)에 다시 답글을 달려는 요청은 거부된다. [env:unit]

- **SC-015** (`NFR-006` 관련): 댓글 목록은 `createdAt` 오름차순으로 반환된다. [env:unit]

## 범위 외

- **댓글 수정·삭제**: 이번 spec 범위에 포함하지 않는다. 댓글은 작성 후 변경·삭제 불가. 향후 별도 spec으로 다룬다.
- **읽음 표시(Read Receipt)**: 댓글별 읽음 상태 추적은 포함하지 않는다.
- **멘션(@mention)**: 댓글 내 특정 사용자 멘션 기능은 포함하지 않는다.
- **파일 첨부**: 댓글에 이미지·파일 첨부는 포함하지 않는다. 텍스트 전용.
- **실시간 업데이트(WebSocket/SSE)**: 댓글 목록의 실시간 반영은 포함하지 않는다. 페이지 새로고침 또는 수동 갱신으로 확인한다.
- **기존 MessageTemplate 연동**: 이번 댓글 스레드는 기존 메시지 템플릿 시스템과 별개로 개발한다. 두 시스템 간 통합은 범위 외.
- **클라이언트 이메일 인증**: 클라이언트가 입력한 이메일 소유권 인증(OTP 등) 절차는 포함하지 않는다.
- **스팸/어뷰즈 방지**: Rate limiting, CAPTCHA 등 스팸 방지 메커니즘은 포함하지 않는다.
- **댓글 알림 ON/OFF 설정**: 기존 `notificationSettings`의 타입별 ON/OFF 제어 확장은 이번 범위에 포함하지 않는다. 댓글 관련 알림은 항상 발송된다.
